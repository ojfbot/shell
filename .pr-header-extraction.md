## Redux store composition options ([shell] #5 disposition)

### Option A — Host-owns-store, props down (current pattern)

`@ojfbot/shell` exports zero Redux artefacts. The shell-app host owns the full Redux store, selects values, and passes them as props to `Header` (and `AppSwitcher`) via thin "Connected" wrapper components that live in shell-app. This is exactly the pattern AppSwitcher uses today.

**Pros:** MF boundary is completely clean — the UI package has no opinion about state management. Components are trivially testable in Storybook with no Provider. Hosts that aren't Redux shops can still adopt the component.

**Cons:** As more stateful components migrate, shell-app accumulates a growing set of `*Connected` wrappers. Not inherently a problem, but the wrapper layer is boilerplate.

### Option B — @ojfbot/shell exports store slices for hosts to compose

The package exports RTK slice reducers and action creators. The host imports and mounts them into its store, then selects from them normally.

**Pros:** Reduces shell-app boilerplate; the slice logic lives in the package next to the component.

**Cons:** Couples the UI package to Redux Toolkit — violates the zero-external-state-management rule for UI packages. Any host not using RTK is blocked. Versioning the slice shape across multiple consumers becomes a coordination problem.

### Option C — @ojfbot/shell provides a React context

The package exports a `ShellProvider` that holds its own state internally. Components consume it via `useShellContext`.

**Pros:** Completely self-contained; no Redux dependency in either the package or the host.

**Cons:** Breaks the single-store principle. The host cannot observe or time-travel-debug internal shell state. Requires a context boundary that may conflict with the host's own Carbon/provider tree.

### Recommendation

**Option A is the correct choice for Phase 1 and should be locked for all stateful component extractions.**

The props-down pattern keeps `@ojfbot/shell` a pure UI library — no state management opinion, maximum portability, and Storybook-first development with no Provider setup. The `*Connected` wrapper layer in shell-app is the appropriate seam for store wiring: it's explicit, easy to grep, and keeps the MF boundary auditable. Options B and C each couple the UI package to an implementation detail that would be expensive to undo once ChatBar and other components land.

**Decision criterion:** This option must be locked before ChatBar or any stateful component extraction begins.
